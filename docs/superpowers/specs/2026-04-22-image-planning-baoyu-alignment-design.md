## 2. baoyu 的真实处理方式

基于已核对的真文档：

- `/Users/frank/Documents/MyStudio/repo/skills/baoyu-skills/skills/baoyu-article-illustrator/SKILL.md`
- `/Users/frank/Documents/MyStudio/repo/skills/baoyu-skills/skills/baoyu-article-illustrator/references/workflow.md`

当前能确认的事实如下：

### 2.1 位置分析先于 Prompt 组装

`baoyu` 先做文章分析，识别值得配图的位置，再生成 `outline.md`。  
后续的生图 Prompt 组装，发生在位置已经确定之后。

顺序是：

1. 分析文章
2. 确认参数
3. 生成 `outline.md`
4. 为每个位置生成结构化 Prompt
5. 出图
6. 插回文章

### 2.2 `Step 2` 的职责

`Step 2` 是内容分析，不是出图。

文档中明确包含：

- 识别内容类型
- 识别核心论点
- 识别哪些位置值得配图
- 识别抽象概念、数据对比、流程
- 排除不该画的内容
  - 不画字面比喻
  - 不画装饰性内容
  - 不画泛化、无信息量的图

### 2.3 `Step 3` 的职责

`Step 3` 不是只确认配图密度。  
它是参数确认阶段，至少包括：

- 图片类型或预设
- 配图密度
- 风格
- 可选的配色

### 2.4 `Step 4` 的产物

`Step 4` 产出的是 `outline.md`，其中每张图都有：

- `Position`
- `Purpose`
- `Visual Content`
- `Filename`

其中 `Position` 使用：

- `section`
- `paragraph`

也就是说，`baoyu` 的中间真源不是“候选锚点列表”，而是“已确定位置的大纲”。

### 2.5 Prompt 不是自然语言拼接，而是结构化模板

`baoyu` 的生图 Prompt 处理方式是：

1. 按图片类型选模板
2. 注入风格规则
3. 注入配色规则
4. 填入文章真实内容
5. 追加全局约束
6. 处理参考图与水印

所以它的 Prompt 层不是简单文本拼接，而是模板化构造。

### 2.6 当前限制

当前在 `baoyu-article-illustrator` 目录内，我只核对到了：

- `SKILL.md`
- `references/workflow.md`
- Prompt 与 style / palette 参考文档

没有在该目录内找到可直接核对的 toolkit 实现代码。  
所以当前对 `baoyu` 的判断，依据是其真文档流程，不是假设它一定存在某种本地解析实现。

## 3. 当前插件现状

当前插件在这两步上与 `baoyu` 不一致。

### 3.1 位置规划现状

当前插件仍然是：

1. 程序先提取文章 block
2. 程序先按本地规则推配图数量
3. 程序先组织候选位置
4. 模型再从候选中选位置

这个问题已经明确：

- 程序承担了过多规划权
- 模型不是主规划者
- 与 `baoyu` 的“先分析，再产出位置大纲”不一致

### 3.2 Prompt 现状

当前插件的 Prompt 仍然只是简单文本拼接，主要包含：

- 标题
- 风格
- 图片类型
- 创意方向
- 位置信息
- 提示意图
- 技术要求
- 负面约束
- 文章摘要

当前缺失：

- 类型专属模板
- 风格规则注入
- 配色规则注入
- 真实内容字段填充
- Prompt 文件落盘
- 参考图 frontmatter
- 模板化的布局 / 区域 / 标签 / 颜色 / 风格 / 比例结构

## 4. 后续要重做的两步主流程

后续只围绕下面两步重做。

### 4.1 第一步：模型先分析文章并产出位置大纲

#### 输入

- 文章全文
- 系统规则
- 用户已提交的配置
  - 配图密度
  - 风格
  - 图片类型或预设

#### 模型职责

- 识别内容类型
- 提取核心论点
- 识别所有值得配图的位置
- 按用户配置筛出最终位置
- 为每个位置给出配图目的
- 产出最终位置大纲

#### 输出

输出应接近 `outline` 中间产物，而不是松散候选集。

每个位置至少包含：

- 位置类型：`section` / `paragraph`
- 位置描述
- 配图目的
- 视觉内容
- 图片类型
- 文件名或稳定标识

#### 输出顺序

`outline` 必须按正文出现顺序返回。  
后续程序、预览和插入逻辑都默认依赖这个顺序，不再额外重排。

#### 关键原则

- 位置由模型先决定
- 程序不再先大致规划位置和数量
- 程序也不再先把候选范围强约束死

#### 缓存要求

4.1 的最终输出需要直接保存到 cache。

当前要求保存的核心内容：

- `articleType`
- `coreArguments`
- `imageCount`
- `outline`

当前阶段不处理：

- cache 失效规则
- cache 命中策略
- 参数变更后的重算策略
- `imageCount` 与 `outline.length` 的额外一致性约束
- 同一位置重复检测规则

也就是说，这一轮只先确定：

- 4.1 的结果会被缓存
- 4.2 后续直接消费这份缓存结果

### 4.2 第二步：程序按大纲为每个位置组装结构化 Prompt

#### 输入

- 第一步产出的最终位置大纲
- 图片类型模板
- 风格规则
- 配色规则
- 文章真实内容

#### 程序职责

- 按图片类型选模板
- 注入风格规则
- 注入配色规则
- 填入文章真实内容
- 追加全局约束
- 生成每个位置对应的结构化 Prompt

#### 输出

每个位置一份结构化 Prompt。  
后续再由生图模型执行。

#### 关键原则

- 这一步只解决“怎么画”
- 不再重新决定“画几张、画在哪”

### 4.4 Prompt 显式注入原则

由于 `baoyu` 原本是 skill 形态，模型在执行前天然可读：

- `SKILL.md`
- `references/workflow.md`
- `references/prompt-construction.md`
- `references/styles/*.md`
- `references/palettes/*.md`

而当前插件是一次次独立调用 LLM。  
因此，凡是会直接影响本次模型判断结果的规则，都不能只留在本地文档或代码里，必须显式写进 `4.1` 或 `4.2` 的 prompt。

#### 必须显式写进 4.1 prompt 的内容

这些属于“位置规划规则”，不能只靠代码暗示：

1. `section / paragraph` 的位置定义  
2. 优先识别规则
   - 核心论点
   - 抽象概念
   - 数据对比
   - 流程 / 工作流
   - 架构 / 框架 / 关系
3. 排除规则
   - 装饰性内容
   - 通用插图
   - 开头 / 过渡 / 结尾 / 号召
   - 比喻只取底层概念，不取字面对象
4. 配图密度规则
   - `minimal`
   - `balanced`
   - `per-section`
   - `rich`
5. 正文图类型规则
   - 用户固定类型如何处理
   - “按文章内容”如何处理
   - `inlineType` 枚举值
6. 输出格式要求
   - JSON 顶层字段
   - `outline` 每项字段
   - 顺序要求
   - `section / paragraph` 的 `excerpt` 约束

#### 必须显式写进 4.2 prompt 的内容

这些属于“Prompt 组装规则”，否则模型无法按模板稳定补全结构化内容：

1. 类型模板骨架
   - `infographic`
   - `scene`
   - `flowchart`
   - `comparison`
   - `framework`
   - `timeline`
2. 风格规则
   - 当前选中风格的渲染要求
   - 线条、纹理、背景、文字、人物表现
3. 配色规则
   - 当前 palette 或默认配色
   - 颜色语义
   - hex 仅作渲染指导，不得画成文字
4. 全局默认规则
   - 留白
   - 简洁构图
   - 禁止写实人脸
   - 文本样式
   - 比例要求
5. 输出格式要求
   - 如果 4.2 返回结构化结果或 Prompt 文件内容，必须给明确 schema 或示例

#### 不需要写进 prompt 的内容

这些保留在程序实现层，不必喂给模型：

1. cache 逻辑
2. 文件路径和目录规则
3. Obsidian 宿主细节
4. 插件 UI 状态
5. 具体插入 Markdown 的实现方式
6. 文件落盘与命名的代码执行流程

### 4.3 Prompt 设计边界

后续记录与实现时，封面和正文图的 Prompt 设计分开处理。

#### 原则

- 封面 Prompt 和正文图 Prompt 分开定义
- 两者字段可以不同
- 两者共享统一风格体系
- 当前优先落正文图 Prompt
- 封面 Prompt 延后单独设计

#### 直接影响

- 后续 Prompt schema 不再强行封面 / 正文共用一套字段
- 正文图这条线先按 `baoyu` 方式重做
- 风格、配色、全局约束仍需保持统一来源，避免封面和正文图完全脱节

## 5. 当前确认的重做方向

后续重做时，主流程应改成：

1. 点击智能配图
2. 模型先分析文章并产出位置大纲
3. 程序按位置大纲组装结构化 Prompt
4. 生图模型执行出图
5. 程序插回 Markdown

也就是说：

- 模型负责“定图”
- 程序负责“写图”和“执行”

### 5.1 4.1 的开发策略

`4.1` 不直接在旧链上改写，先采用并行新实现。

原则如下：

- 先新增一套 `4.1` 专用代码文件
- 旧 `4.1` 链路先保留，不继续扩展
- 新链只负责：
  - 单次 LLM 交互
  - 输出最终正文图位置大纲
  - 写入新 cache
- 等新 `4.1` 跑通并完成测试切换后，再统一删除旧链

这样做的目的：

- 避免一边删旧逻辑一边丢上下文
- 降低重构期间智能配图整体不可用的风险
- 让新旧行为可以对照验证

### 5.2 4.1 完成后的废弃说明

`4.1` 新链开发完成、测试切换完成后，需要额外输出一份废弃说明文档。

这份文档必须记录：

1. 已废弃文件
2. 已废弃字段
3. 已废弃描述
4. 替代关系
5. 删除时机

这份文档是交付物的一部分，不是可选补充。

目的：

- 避免旧代码、旧字段、旧描述残留
- 让后续删除动作有明确清单
- 防止新旧链并存后长期遗留冗余

## 7. 4.1 用的 system prompt 初稿

下面这份 prompt 对应 4.1：

- 只负责“模型先分析文章并产出正文图位置大纲”
- 不负责最终生图 Prompt 组装
- 不依赖账户配置

```text
你是一个微信公众号文章配图规划助手。

你的任务不是总结全文，而是直接为这篇文章产出“正文图位置大纲”。

你需要判断：
- 哪些位置最值得配图
- 为什么值得配图
- 适合什么图片类型
- 这张图应该表达什么

## 输入

你会收到以下信息：
- 文章全文
- 用户当前选择的参数：
  - 配图密度
  - 风格
  - 图片类型
  - 创意方向

## 工作规则

### 1. 先理解文章

你必须先完成以下分析：
- 判断文章内容类型，例如：技术拆解、方法论、观点分析、案例分析、叙事说明
- 提取 2-5 个核心论点
- 找出文章中真正值得配图的位置

### 2. 什么位置值得配图

优先选择以下内容：
- 核心论点承载段落
- 抽象概念解释段落
- 流程步骤段落
- 架构关系段落
- 对比差异段落
- 时间演化段落
- 数据、指标或复杂机制解释段落
- 读者容易产生理解断层的复杂段落

### 3. 什么位置不要配图

不要选择以下内容：
- 开头寒暄、引入、铺垫
- 结尾总结、号召、态度表达
- 纯情绪段落
- 纯修辞段落
- 过渡段
- 没有信息增量的段落

### 4. 排除规则

禁止把以下内容作为主要配图目标：
- 字面比喻
- 装饰性图
- 空泛概念图
- 与正文无强关联的场景图

如果文章使用比喻，你应该表达比喻背后的真实结构或概念，而不是画字面内容。

### 5. 参数使用方式

你必须结合用户参数做判断：

- 配图密度：
  决定最终保留多少个位置

- 风格：
  影响视觉表达倾向，但不改变文章核心逻辑

- 图片类型：
  影响你优先寻找什么样的位置
  例如：
  - 流程图更偏流程段
  - 对比图更偏差异段
  - 框架图更偏结构段

- 创意方向：
  影响这张图应该强调什么

### 6. 输出要求

你必须直接输出“最终正文图位置大纲”，不是候选列表，不是分析笔记。

每一项必须包含：
- positionType
  - section 或 paragraph
- locationText
  - 位置描述，必须让后续程序能据此定位
- excerpt
  - 原文摘录，尽量保持原句可定位
- sectionTitle
  - 所属章节标题，没有则留空
- purpose
  - 为什么这里需要图
- inlineType
  - infographic / scene / flowchart / comparison / framework / timeline
- visualSummary
  - 这张图应该表达什么，不是最终 Prompt

### 7. 输出约束

- 只输出最终结果
- 不输出思考过程
- 不输出候选淘汰过程
- 不输出多余解释
- 不编造文章中不存在的信息
- 所有位置必须能在原文中找到依据

## 输出格式

使用 JSON 输出，格式如下：

{
  "articleType": "文章类型",
  "coreArguments": [
    "核心论点 1",
    "核心论点 2"
  ],
  "outline": [
    {
      "positionType": "section 或 paragraph",
      "locationText": "位置描述",
      "excerpt": "原文摘录",
      "sectionTitle": "章节标题",
      "purpose": "为什么这里需要图",
      "inlineType": "framework",
      "visualSummary": "这张图应该表达什么"
    }
  ]
}
```

这些内容后续再单独规划。

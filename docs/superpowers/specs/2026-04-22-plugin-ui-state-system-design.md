# Plugin UI State System Design

日期：2026-04-22

## 1. 目标

为 `wechat-article-obsidian` 建立一套统一的插件级交互状态系统，覆盖设置页与 workbench 中已经存在的核心控件，解决当前以下问题：

- 同类控件在不同区域状态表现不一致
- `hover / active / selected / disabled / focus` 缺失或表现混乱
- 设置页与 workbench 不是同一套视觉语言
- 卡片、tab、按钮、输入控件的层级和间距不稳定

本次设计保持以下边界：

- 只重构 UI 交互状态与样式系统，不改业务逻辑
- 不改 workbench 信息架构，不把插件做成第二编辑器
- 不引入新的动效系统，只做轻量状态反馈
- 整体风格克制，但保留当前紫色主按钮和选中强调

## 2. 范围

### 2.1 本期覆盖

设置页：

- 顶部 tab
- 折叠卡片
- 账户摘要卡片
- 行内动作按钮
- 输入框 / 文本域 / 下拉 / 开关

workbench：

- 顶部作者标签
- 发布按钮
- 图片操作按钮
- 主题区控件

### 2.2 非目标

- 不改布局结构和信息架构
- 不新增新的业务组件
- 不做整站换肤或多主题切换
- 不重写 Obsidian 原生全局样式
- 不覆盖当前需求未涉及的历史页面或无关实验样式

## 3. 视觉方向

整体风格采用“克制基底 + 紫色强调”的混合方案：

- 基底：白底、浅灰层级、低饱和边框、深灰文字
- 强调：只在主按钮、选中态底线、关键选中态上使用紫色
- 层级：外层白卡片，内层信息项浅灰底
- 状态反馈：清晰但轻量，不使用厚阴影、重渐变或过强动画

这意味着插件要更像一个稳定的 Obsidian 工具台，而不是一个高饱和的独立 Web App。

## 4. 全局状态模型

所有本期覆盖控件统一支持以下状态语义：

- `default`
- `hover`
- `active`
- `selected`
- `focus`
- `disabled`

### 4.1 状态定义

`default`

- 基础可交互样式
- 白底或浅灰底
- 文字和边框保持正常对比

`hover`

- 背景轻微加深
- 边框或文字对比略提升
- 不改变控件尺寸，不引起跳动

`active`

- 比 `hover` 再压一层
- 主要用于按钮、折叠头、可点击卡片

`selected`

- 以紫色作为强调信号
- tab 使用底部紫线
- 选择器、次级按钮、作者标签使用轻紫边框或轻紫底
- 不允许所有组件都变成紫色大面积填充

`focus`

- 统一 focus ring
- 不依赖浏览器默认蓝框
- 仅对键盘可达控件生效

`disabled`

- 降低文字、边框、背景对比
- 移除 hover / active 反馈
- 鼠标指针不再显示为可点击

### 4.2 状态 token

本期不引入新的主题系统，但需要在插件样式中定义一层可复用 token，至少包括：

- 背景：
  - `--wao-surface-base`
  - `--wao-surface-muted`
  - `--wao-surface-hover`
  - `--wao-surface-active`
- 边框：
  - `--wao-border-default`
  - `--wao-border-strong`
  - `--wao-border-selected`
- 文字：
  - `--wao-text-primary`
  - `--wao-text-secondary`
  - `--wao-text-disabled`
- 强调：
  - `--wao-accent`
  - `--wao-accent-muted`
  - `--wao-accent-active`
- 状态：
  - `--wao-success`
  - `--wao-danger`
  - `--wao-focus-ring`

这些 token 只服务当前插件，不试图抽象成 Obsidian 全局设计系统。

## 5. 组件规范

## 5.1 顶部 Tab

适用范围：

- 设置页顶部 `账户 / 模型`

样式要求：

- 默认：浅灰外层容器，tab 本体白底或透明底，深灰字
- hover：tab 背景微变深
- active：较 hover 更深一层
- selected：
  - 文字变深
  - 底部出现紫色选中线
  - 不使用整块高饱和紫底
- disabled：灰字，无底线，无 hover 响应

交互要求：

- tab 的选中态必须可稳定区分，不能只靠背景微差

## 5.2 折叠卡片

适用范围：

- 设置页大类折叠区
- 弹窗中的折叠区

结构要求：

- 外层白底卡片
- 内层内容区使用浅灰底 item
- 卡片头与内容区之间留出明确层级

间距要求：

- 卡片外边距、内边距统一
- 展开后每个 item 的上下左右间距固定
- 不允许字段区域出现“裸露文本域贴边”的状态

状态要求：

- 头部支持 `default / hover / active / selected / disabled`
- 展开状态通过箭头方向和头部背景共同表达

## 5.3 账户摘要卡片

适用范围：

- 设置页账户列表

结构要求：

- 一行展示作者、公众号账号、公众号状态、动作按钮
- 状态点：
  - 绿点：已配置公众号
  - 红点：未配置公众号
- 卡片本身为白底
- 动作按钮为中性次级按钮

状态要求：

- 卡片支持 hover
- 行内按钮支持 default / hover / active / disabled
- `凭证` 按钮可以带状态语义，但不抢主按钮视觉权重

## 5.4 作者标签

适用范围：

- workbench 顶部作者选择控件

样式要求：

- 默认为中性白底或浅灰底触发器
- hover 显示浅灰背景
- active 再压深一层
- selected 使用轻紫边框或轻紫背景
- disabled 显示不可交互态

要求：

- 多账户时箭头可见
- 单账户或无账户时仍需具备明确的状态反馈

## 5.5 按钮体系

分为三类：

- 主按钮：如发布按钮
- 次级按钮：如图片操作按钮、账户卡片行内按钮
- 危险按钮：如删除操作

主按钮：

- 保留紫色主填充
- hover 稍亮
- active 稍深
- disabled 明显降饱和

次级按钮：

- 默认白底或浅灰底
- hover 背景加深
- selected 时才允许引入轻紫强调

危险按钮：

- 不默认大面积红底
- 仅在 hover / active 或明确危险场景中增强危险语义

## 5.6 表单控件

适用范围：

- 输入框
- 文本域
- 下拉
- 开关

样式要求：

- 默认：白底，细边框
- hover：边框增强
- focus：统一紫色 ring
- disabled：灰底灰字灰边

文本域要求：

- 与输入框属于同一控件家族
- 间距、边框、焦点态一致

## 5.7 主题区控件与图片操作按钮

适用范围：

- workbench 主题切换区
- 图片生成、重生成、删除等操作按钮

要求：

- 归入次级按钮体系
- 非选中态保持克制
- 选中或当前态才使用轻紫强调
- 不允许不同操作按钮拥有彼此冲突的 hover / active 逻辑

## 6. 实现边界

## 6.1 样式组织

样式应继续维护在真实源文件中，再由构建脚本同步到插件产物，不能直接改生成后的根样式文件。

本期允许：

- 在现有样式文件中新增 token
- 对设置页和 workbench 样式按组件重组

本期不做：

- 大规模 CSS 架构重写
- 引入新的 CSS-in-JS 或运行时样式方案

## 6.2 选择器策略

优先使用当前插件自己的 class 命名空间，避免污染 Obsidian 全局组件。

要求：

- 不覆盖 `.mod-cta` 等全局类的默认行为
- 只在插件自己的容器作用域内定义状态样式

## 7. 验证要求

实现完成后至少要人工验证以下状态：

- 设置页顶部 tab 的 default / hover / active / selected / disabled
- 折叠卡片头部的 default / hover / active / expanded
- 账户卡片行内按钮的 default / hover / active / disabled
- 作者标签的 default / hover / active / selected / disabled
- 发布按钮的 default / hover / active / disabled
- 图片操作按钮的 default / hover / active / selected / disabled
- 输入框 / 文本域 / 下拉的 default / hover / focus / disabled

如果存在无法进入的 `disabled` 状态，需要通过临时测试数据或开发开关确认样式表现，而不是直接跳过。

## 8. 风险与取舍

风险：

- 这轮覆盖范围已扩大为全插件核心交互控件，如果边改边追加，会反复返工
- 状态 token 如果定义得过多，会引入新的维护负担
- 如果同时改布局，会让问题定位变得困难

取舍：

- 本期只统一交互状态和层级表现，不趁机改布局结构
- 紫色只保留在主按钮和选中强调，不扩散到所有可点击元素
- 优先修正设置页和 workbench 的一致性，不扩成完整设计系统项目

## 9. 完成标准

满足以下条件才算完成：

- 设置页与 workbench 的核心控件状态逻辑统一
- tab、折叠卡片、按钮、输入控件、作者标签在所有关键状态下都有明确视觉差异
- 白卡片 + 灰内容层 + 紫色强调的风格在两个区域都成立
- 没有通过修改业务逻辑来“伪造状态”
- 构建后的插件样式真实生效，并能在 Obsidian 中看到一致结果

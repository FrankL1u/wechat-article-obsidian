# WeChat Article Obsidian Plugin V1 设计方案

日期：2026-04-20

## 1. 目标

构建一个 Obsidian 前端组件化插件，作为当前打开文档的公众号工作台使用。

V1 只解决四件事：

- 基于当前打开的 Markdown 文档实时生成公众号 HTML 预览
- 切换整篇排版主题，并保证主题输出与现有 `ls-wechat-article` 能力一致
- 基于现有图片规则生成封面和正文配图，并支持单张重生成
- 输出一篇新的 Markdown 文章，自动在 Obsidian 中打开

## 2. 非目标

以下能力不进入 V1：

- 热点抓取
- TrendRadar MCP 接入
- 知识库选题和正文生成
- 微信发布
- 学习回写
- 在工作台中编辑正文
- 回写并覆盖原始文档
- 自由 WYSIWYG 富文本排版编辑器
- 局部样式自定义

## 3. 产品定义

### 3.1 输入源

- 输入源固定为当前打开的 Obsidian Markdown 文档
- 工作台实时读取当前编辑器内容
- 读取内容包含尚未保存到磁盘的修改
- 原始文档始终保持不变，不被插件覆盖

### 3.2 工作台形态

- 插件在左侧工具栏注册一个 icon
- 点击 icon 后，在右侧打开自定义 `ItemView`
- 工作台仅承担预览、主题切换、图片生成、结果输出职责
- 正文编辑仍在 Obsidian 原生编辑器中完成

### 3.3 工作台能力

- 实时 HTML 预览
- 整篇主题切换
- 封面和正文配图参数选择
- 一键生成图片
- 单张图片重生成
- 生成新文章

### 3.4 输出结果

- 最终输出为一篇新的 `.md` 文档
- HTML 仅用于工作台预览，不作为正式产物
- 新生成的 Markdown 文档会自动在 Obsidian 中打开

## 4. 用户流程

### 4.1 核心流程

1. 用户在 Obsidian 中打开一篇 Markdown 文档
2. 用户点击左侧插件 icon，打开右侧工作台
3. 工作台实时读取当前文档内容并渲染公众号预览
4. 用户选择主题，预览即时刷新
5. 用户配置图片参数：
   - 封面类型
   - 风格
   - 创意方向
   - 配图数量
6. 用户点击生成图片
7. 插件根据现有 skill 规则：
   - 生成封面
   - 判断正文配图位置
   - 生成正文配图
8. 预览中显示封面与正文配图结果
9. 用户可对单张图片执行重生成
10. 用户点击“生成新文章”
11. 插件产出新的 Markdown 文档及其引用图片资源
12. 插件自动在 Obsidian 中打开新文档

### 4.2 预览行为

- 封面默认显示在预览顶部
- 正文配图按现有 skill 规则推断的位置显示
- 主题切换只影响预览渲染结果，不改变正文内容本身
- 工作台不保存正文编辑状态，因为正文不在工作台编辑

## 5. 主题与预览设计

### 5.1 主题单一来源

- 预览主题以 `builtin-themes.json` 为单一来源
- 插件不得重新维护另一套主题定义
- 插件只负责：
  - 主题选择 UI
  - 当前主题状态管理
  - 将主题应用到预览渲染流程

### 5.2 复用范围

- 预览 HTML 生成逻辑沿用现有 `ls-wechat-article` 的 toolkit 能力
- 主题解析沿用现有主题数据和引擎能力
- 插件工作台自身 UI 样式独立实现，不与公众号主题 CSS 混用

### 5.3 预览容器

- 预览区采用隔离容器实现，优先选 `iframe`
- 采用隔离容器的原因：
  - 避免公众号文章 CSS 污染工作台控件
  - 避免 Obsidian 主题样式影响文章预览
  - 降低调试复杂度

## 6. 图片系统设计

### 6.1 图片规则来源

- 图片位置判断规则沿用现有 `ls-wechat-article` 规则
- 插件不要求用户手工选择正文配图插入位置
- 插件不重新定义正文配图规划算法

### 6.2 用户可配置参数

用户在生成图片前可设置：

- `cover_type`
- `style`
- `creative_direction`
- `数量`

这些参数在 UI 中全部使用中文名称展示。

### 6.3 图片生成行为

- 点击生成图片后，插件先基于当前文档与参数生成一组结果
- 封面与正文配图先进入当前工作流结果，不直接改原文
- 用户可对任意单张图片执行重生成
- 重生成只替换当前工作流中的该图片结果

### 6.4 图片输出策略

- 当用户点击“生成新文章”时，使用当前确认过的图片结果
- 插件将图片资源落盘到 Vault 内正式资源目录
- 新文章 Markdown 引用这些正式资源

## 7. 输出与目录规则

### 7.1 新文章输出

- 输出文件格式固定为 Markdown
- 新文章命名规则：
  - `原文件名-wechat-YYYYMMDD-HHmm.md`

示例：

- `原文-wechat-20260420-1430.md`

### 7.2 输出目录设置

设置中增加“输出目录”：

- 默认关闭时：输出到原文同目录
- 开启后：允许指定输出目录
- 支持两种形式：
  - 相对路径：相对于知识库根目录
  - 绝对路径：仅允许解析到知识库内部有效路径

V1 不支持输出到 Vault 外任意目录，原因如下：

- 生成后需要自动在 Obsidian 中打开
- Obsidian 对 Vault 内文件的打开和管理最稳定
- Vault 外文件会引入产品能力与宿主能力不一致的问题

### 7.3 资源目录规则

图片正式资源落到 Vault 根目录下：

- `.wechat-article-obsidian/`

目录规则为：

- `年-月-时间戳`

示例：

- `.wechat-article-obsidian/2026-04-1713765400000/`

## 8. 存储分层

### 8.1 插件内部数据

插件目录下使用：

- `.userdata`

用于保存：

- 插件设置
- 运行缓存
- 中间结果
- 临时图片
- 当前工作流状态

### 8.2 Vault 内正式资产

Vault 根目录下使用：

- `.wechat-article-obsidian`

用于保存：

- 最终新文章所引用的正式图片资源

原则：

- `.userdata` 是插件私有运行区
- `.wechat-article-obsidian` 是知识库内长期可引用资产区

## 9. 设置页设计

### 9.1 V1 设置范围

V1 设置页只放当前真正生效的配置，不提前暴露二期能力。

V1 设置项包括：

- 图片服务商
- API Key
- Base URL
- Model
- 默认封面类型
- 默认风格
- 默认创意方向
- 输出目录开关
- 输出目录路径

### 9.2 文案要求

- 所有设置项使用中文名称
- 后续工作台页面元素统一使用中文名称
- 不在 V1 暴露以下配置：
  - MCP 配置
  - 文本生成模型配置
  - 微信账户配置
  - 学习相关配置

## 10. 页面结构

### 10.1 工作台页面分区

建议页面结构如下：

- 顶部工具栏
  - 当前文档名
  - 当前主题
  - 生成图片
  - 生成新文章
- 参数区
  - 封面类型
  - 风格
  - 创意方向
  - 配图数量
- 预览区
  - 公众号 HTML 预览
- 图片结果区
  - 封面卡片
  - 正文图片卡片
  - 单张重生成入口
- 状态区
  - 生成中
  - 错误信息
  - 输出结果提示

### 10.2 交互原则

- 工作台不提供正文编辑框
- 工作台不提供原文覆盖保存入口
- 工作台始终围绕“预览和生成新文章”组织交互

## 11. 技术分层

### 11.1 代码分层

推荐结构：

- `src/main.ts`
  - 插件入口
  - 注册 icon
  - 注册 `ItemView`
  - 注册设置页
- `src/obsidian/*`
  - Obsidian 宿主层
  - 当前文件读取
  - 当前编辑器监听
  - 新文档写入
  - 设置存储
- `src/ui/*`
  - 工作台 UI 组件
  - 可在普通 web 环境中独立运行
- `src/core/*`
  - 共享核心逻辑
  - 主题解析
  - Markdown 转预览 HTML
  - 图片规划
  - 图片生成
  - Markdown 输出组装
- `src/services/*`
  - 输出目录解析
  - 文件命名
  - 资源落盘
  - 工作流状态组装

### 11.2 硬约束

- 工作台 UI 必须可脱离 Obsidian 独立运行
- Obsidian API 不直接写入纯 UI 组件
- 业务逻辑不堆进 `ItemView`

## 12. 开发与测试策略

### 12.1 双入口开发

采用双入口：

- `web dev` 入口
  - 本地起前端服务
  - 调试 UI、预览、状态流
- `obsidian plugin` 入口
  - 用于最终集成测试

### 12.2 测试分层

#### 纯逻辑测试

可脱离 Obsidian 进行单测：

- 输出目录解析
- 文件名生成
- 资源路径规则
- frontmatter 组装
- Markdown 输出拼装
- 预览输入数据整形

#### UI 测试

在普通前端环境中调试：

- 主题切换
- 参数面板
- 图片卡片
- 预览布局
- 空态、错误态、加载态

#### Obsidian 集成测试

必须进入 Obsidian 验证：

- 左侧 icon
- 右侧 `ItemView`
- 当前文档变化监听
- 读取未保存编辑内容
- 生成新文章后自动打开
- 设置存取

## 13. 风险与约束

### 13.1 主要风险

最大的风险不是 UI，而是复用现有 skill 能力的方式。

禁止方案：

- 直接依赖本地 skill 仓库路径
- 直接依赖某个开发机上的私有目录结构

推荐方案：

- 把 V1 真正需要的能力提炼进插件项目自己的共享核心模块
- 至少抽出：
  - 主题数据与主题引擎
  - Markdown 预览转换
  - 图片规划逻辑
  - 图片生成调用层

### 13.2 设计约束

- V1 必须严格收敛，不提前做二期功能壳子
- 工作台不进入正文编辑职责
- 原文不覆盖
- 生成结果以新文章为准

## 14. V1 完成标准

满足以下条件即可视为 V1 完成：

- 能从当前打开文档实时渲染公众号 HTML 预览
- 能切换主题，且主题结果与现有 toolkit 一致
- 能设置图片参数并生成封面与正文配图
- 能对单张图片执行重生成
- 能将当前结果输出为新的 Markdown 文章
- 能自动在 Obsidian 中打开新文章
- 能通过设置页配置图片生成和输出目录

## 15. 后续扩展位

二期能力在架构上预留，但不进入 V1：

- 热点抓取
- TrendRadar MCP
- 知识库选题
- 正文生成
- 微信发布
- 学习回写

V1 的架构目标不是一次做完，而是保证这些能力后续能加进去而不推翻当前分层。
